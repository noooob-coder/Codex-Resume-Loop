# Python / Go / Rust 实现注意点

本文件用于在代码任务中提供按语言分类的默认实现建议。它不是强制规范；如果用户明确要求不同风格或为了兼容现有项目需要遵循其他约定，以用户或项目约定为准。

## Python

### 适用场景速记
- 适合脚本、自动化、数据处理、胶水层、原型验证、Web 后端。
- 当任务 I/O 密集、开发速度优先、生态依赖强时，优先考虑 Python。
- 当任务 CPU 密集、低延迟或高并发要求明显时，要主动提醒用户 Python 可能不是最优语言，或至少要规避明显低效实现。

### 容器与数据结构选择
- 顺序追加、遍历：优先 `list`
- 头部弹出频繁：用 `collections.deque`，不要反复 `list.pop(0)`
- 成员判断 / 去重：优先 `set`
- 键值映射 / 计数：优先 `dict` / `collections.Counter`
- 优先队列：`heapq`
- 默认字典：`collections.defaultdict`
- 固定字段的小对象：优先 `dataclasses.dataclass`，而不是随意挂属性

### 性能与实现习惯
- 避免在循环里做重复的字符串拼接；优先收集后 `"".join(...)`
- 避免在热路径里重复正则编译、JSON 解析、日期解析、文件打开
- 大文件优先流式处理，不要一次性全量读入内存，除非数据规模明确可控
- 不要在循环里频繁深拷贝、切片复制或构造大临时列表；必要时用生成器
- 明确区分 `list` 推导式和生成器表达式：需要惰性时优先生成器
- 对于高频成员判断，不要在 `list` 上反复 `in`；改成 `set`
- 排序只做一次；不要在循环内重复排序同一批数据
- 如果只需要前 k 个元素，优先考虑 `heapq.nsmallest` / `nlargest` 或维护小根堆

### 可读性与工程习惯
- 函数保持单一职责，避免把解析、校验、计算、格式化揉在一个函数里
- 使用类型注解，尤其是公共函数、边界层、数据结构
- 优先使用上下文管理器处理文件、锁、连接等资源
- 异常只捕获真正能处理的类型；避免裸 `except:`
- 日志使用 `logging`，不要保留 `print` 调试输出
- 配置走参数、配置文件或环境变量，不要硬编码路径、token、host

### 并发与异步
- I/O 密集并发优先考虑 `asyncio` 或线程池
- CPU 密集任务不要盲目用线程期待提速；要提醒 GIL 的影响，必要时用多进程或换语言/原生扩展
- 使用 `asyncio` 时，不要在协程里夹杂阻塞 I/O；阻塞调用要下沉到线程池或替换成异步实现

### 常见踩坑提醒
- 不要把可变对象当默认参数，例如 `def f(x=[])`
- 注意闭包捕获循环变量的问题
- 需要复制嵌套结构时区分浅拷贝与深拷贝
- 时间处理尽量使用时区明确的对象，避免 naive datetime 在边界场景出错
- 序列化 / 反序列化时注意编码统一为 UTF-8

### 默认自检项
- 是否把频繁成员判断从 `list` 改成了 `set`
- 是否避免了热路径上的重复解析、重复排序、重复 I/O
- 是否移除了调试 `print`、未使用 import、临时脚本
- 是否为公开函数补了必要的类型注解和异常说明

## Go

### 适用场景速记
- 适合网络服务、并发 I/O、CLI 工具、服务端基础设施、批处理管道。
- 当需要简单部署、稳定并发、较低内存开销和明确性能时，优先考虑 Go。

### 容器与数据结构选择
- 顺序数据：`slice`
- 键值映射 / 集合：`map[K]V`；集合通常用 `map[T]struct{}`
- 队列：可用 `slice` 维护头尾索引，避免频繁整体移动
- 堆 / 优先队列：`container/heap`
- 排序：`sort.Slice`
- 结构化数据优先 `struct`，不要为了省事过度使用 `map[string]any`

### 性能与实现习惯
- 已知规模时优先预分配：`make([]T, 0, n)`、`make(map[K]V, n)`
- 注意 `append` 扩容成本；在热点路径上尽量预估容量
- 构造大字符串优先 `strings.Builder` 或 `bytes.Buffer`
- 避免在循环中频繁做 `fmt.Sprintf` 这类较重操作；必要时改用更直接的拼接或预格式化
- 大对象不要无意义拷贝；根据语义考虑传指针还是值，但不要为了“性能”无脑全用指针
- 对字节流、文件、网络数据优先流式处理，避免整块读入
- 避免过早抽象；先把数据流和错误路径写清楚，再提炼复用

### 错误处理与资源管理
- 错误要就近处理并带上下文，必要时用 `fmt.Errorf("...: %w", err)` 包装
- 获取资源后尽快 `defer Close()`，但也要注意在超长循环里滥用 `defer` 可能积压资源
- 不要吞掉错误；如果忽略错误，必须有明确理由
- 公共函数返回值要稳定清晰，避免返回过多“魔法状态”布尔值

### 并发习惯
- 用 goroutine 做并发，但不要为每个小任务无节制起 goroutine
- channel 用于通信和协调，不要把它当万能队列到处套
- 共享状态需要明确同步：`sync.Mutex` / `sync.RWMutex` / `atomic`
- 任务取消、超时、请求范围控制优先使用 `context.Context`
- 注意 goroutine 泄漏：发送/接收双方都要有退出路径

### 常见踩坑提醒
- `for range` 变量复用导致闭包拿到同一个变量值
- `slice` 共享底层数组，截取后修改可能互相影响
- `nil` slice 与空 slice 的序列化行为可能不同
- map 并发读写不安全，不能侥幸依赖“看起来没事”
- 接口值与 `nil` 的比较可能产生直觉外行为

### 默认自检项
- 是否为热点 `slice` / `map` 做了合理预分配
- 是否避免了不必要的 `fmt.Sprintf`、对象拷贝、全量读入
- 是否检查了 goroutine 生命周期、context 传递、资源关闭
- 是否移除了未使用变量、调试日志、废弃分支

## Rust

### 适用场景速记
- 适合系统工具、性能敏感组件、并发程序、CLI、服务端热点模块、需要内存安全的底层逻辑。
- 当任务要求高性能、低开销、强类型和更严格正确性时，优先考虑 Rust。

### 容器与数据结构选择
- 顺序数据：`Vec<T>`
- 双端队列：`VecDeque<T>`
- 键值映射：`HashMap<K, V>`；有序映射：`BTreeMap<K, V>`
- 集合：`HashSet<T>` / `BTreeSet<T>`
- 共享所有权：优先先想办法避免；确有必要再考虑 `Rc` / `Arc`
- 可变共享状态：`Mutex<T>` / `RwLock<T>`，只在确实需要时使用

### 性能与实现习惯
- 已知规模时优先 `Vec::with_capacity`、`HashMap::with_capacity`
- 避免无意义 `clone()`；先判断能否借用 `&T` / `&mut T`
- 字符串处理注意 `String` 与 `&str` 的边界，避免不必要分配
- 在热路径上避免反复分配临时对象；必要时复用 buffer
- 对迭代器链保持适度：清晰时很好，用到晦涩影响维护时可落回显式循环
- 错误处理优先 `Result<T, E>`，让错误类型和上下文清楚
- 公共 API 尽量设计成借用优先、所有权转移明确

### 所有权与生命周期习惯
- 先写清数据所有权，再写代码；不要在借用报错后用大量 `clone` 硬压过去
- 函数参数优先考虑 `&str`、`&[T]` 等借用形式，提升复用性与减少分配
- 返回引用时要确保生命周期真的表达了业务关系；如果解释不清，考虑返回拥有所有权的数据
- 将复杂借用冲突拆成更小作用域、辅助函数或中间变量

### 并发习惯
- 线程间共享优先 `Arc<T>` + 同步原语，而不是绕过类型系统
- 异步代码中注意 `Send` / `Sync` 约束；不要混入阻塞操作卡住 runtime
- 对高吞吐网络/IO 服务，优先说明使用的 runtime（如 Tokio）及其约束

### 常见踩坑提醒
- 为了“先编过”而滥用 `clone()`、`unwrap()`、`expect()`
- 在错误边界或外部输入上直接 `unwrap()` 很危险
- 过度嵌套的泛型/trait bound 会显著损害可读性
- `HashMap` / `Vec` 扩容与重新分配在热点场景仍要关注
- 字符串按字节和按字符边界不同，切片要小心 UTF-8

### 默认自检项
- 是否去掉了可避免的 `clone()`、`unwrap()`、`expect()`
- 是否给主要容器做了合理的 capacity 预估
- 是否优先使用借用参数而非强制所有权转移
- 是否清理了未使用模块、调试输出、临时辅助代码

## 跨语言默认比较规则
- 先遵循用户指定语言；不要擅自换语言
- 再遵循项目现有语言和生态；不要为了局部优化破坏一致性
- 如果用户没有指定语言，可根据任务特征做简短建议：
  - 开发效率 / 脚本 / 自动化优先：Python
  - 服务端并发 / CLI / 易部署优先：Go
  - 极致性能 / 内存安全 / 底层工具优先：Rust
- 给出代码时，默认使用该语言社区主流风格，而不是机械套同一套命名或写法
